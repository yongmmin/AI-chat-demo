# AI 도구 사용 내역

## 사용한 AI 도구

| 도구 | 용도 | 사용 비중 |
|------|------|----------|
| Claude Code — Opus 4.6 | 핵심 설계, 복잡한 로직 구현, 아키텍처 결정 | 주력 |
| Claude Code — Sonnet 4.6 | 빠른 반복 작업, CSS 수정, 소규모 기능 추가 | 보조 |

> 작업 성격에 따라 모델을 전환하며 사용했다. 설계·디버깅처럼 추론이 필요한 작업은 Opus 4.6, 반복적인 스타일 조정이나 단순 수정은 Sonnet 4.6을 사용했다.

## AI에게 위임한 작업

### 서버 버그 분석 및 수정
- **dotenv 로드 순서 버그**: `server/stream.ts`에서 `OPENROUTER_API_KEY`를 모듈 최상위 레벨에서 캡처하고 있었음. ES 모듈 특성상 `import` 체인(`index.ts → routes.ts → stream.ts`)은 `dotenv.config()` 실행 전에 평가되므로 환경변수가 항상 `undefined`로 읽히는 버그였음. 환경변수를 함수 호출 시점에 읽도록 수정.
- **테스트 타임아웃 가드**: 무료 OpenRouter 모델의 응답 지연으로 SSE 테스트가 간헐적으로 타임아웃되는 문제. `NODE_ENV === "test"` 시 OpenRouter를 비활성화하고 시나리오 스트림을 강제 사용하도록 `hasOpenRouterKey()`에 가드 추가.

### 전체 레이아웃 및 디자인
- **2패널 구조 설계**: 좌측 채팅 + 우측 아티팩트 프리뷰 슬라이드 패널. artifact_delta 수신 시 패널이 슬라이드로 열리는 UX 결정
- **사이드바 디자인**: 열림 상태(세션 목록 + 푸터)와 닫힘 상태(아이콘만 있는 미니바) 두 가지 상태, 토글 트랜지션
- **색상 시스템**: 포인트 컬러 `#6c5ce7`(보라), 텍스트 계열 `#101828`/`#667085`, 배경 `#f9fafb`, 보더 `#e4e7ec`로 일관된 팔레트 구성
- **반응형 설계**: ≤560px 사이드바 오버레이, ≤900px 프리뷰 스택형 전환, 901–1024px 프리뷰 축소(320px) 등 3단계 브레이크포인트

### ChatSessionPage 핵심 구현
- **`useChatStream` 훅 설계**: SSE(`EventSource`) 생성·구독·정리 로직을 훅으로 캡슐화. `text_delta` / `artifact_delta` / `done` 이벤트 각각 핸들링. malformed JSON(서버가 의도적으로 10% 확률로 주입) 방어 처리
- **타이핑 애니메이션**: 수신 텍스트를 `bufferRef`에 누적하고, `displayLenRef`를 `setTimeout` 루프로 증가시켜 한 글자씩 표시. 버퍼가 20자 이상 밀리면 3자씩 빠르게 따라잡는 배속 로직 포함
- **스트리밍 동기화**: `done` 이벤트 수신 후 `streamText` 초기화 + `loadSession()` 재호출로 서버에 저장된 최종 메시지로 교체. 초기 구현에서 누락되어 있었으나 직접 발견 후 추가 지시
- **Optimistic UI**: 메시지 전송 즉시 임시 user 메시지를 로컬에 추가하고, 스트리밍 완료 후 서버 응답으로 교체
- **에러 처리**: `getSession` 실패 시 에러 메시지 + "다시 시도" 버튼으로 재시도 가능

### 보너스 기능
- **답변 스타일 선택기**: 서버 `Session`에 `style` 필드 추가, 메시지 POST 시 세션에 저장, `streamFromOpenRouter`에서 `buildSystemPrompt(session.style)`로 system prompt 분기. 클라이언트는 헤더 우측 `<select>` UI, 미선택 시 style 미전송(기존 테스트 호환 유지)
- **마크다운 렌더링**: `react-markdown` + `remark-gfm` + `rehype-highlight` 도입. assistant 메시지와 스트리밍 텍스트 모두 실시간 마크다운 렌더링
- **복사 버튼**: `ArtifactPreview` 헤더에 `navigator.clipboard.writeText()` 기반 복사 버튼, 2초 후 "복사됨 ✓" 피드백

## AI 제안을 수정하거나 거절한 경우

| 상황 | AI 제안 | 내 판단 | 이유 |
|------|---------|---------|------|
| 스트리밍 완료 후 동기화 누락 | 스트리밍 `done` 후 `streamText`만 표시하고 세션 리로드 없이 종료 | 버그 직접 발견 후 수정 지시 | 실제 사용 중 스트리밍 완료 후 서버 메시지와 클라이언트 상태가 불일치하는 것을 발견. `done` 이후 `loadSession()` 재호출로 동기화하도록 지시 |
| 답변 스타일 기능 아키텍처 | 클라이언트에서 message content에 style prefix 삽입 방식 언급 | 서버 수정 방식으로 방향 제시 | loadSession 후 prefix가 채팅에 그대로 노출되는 문제를 먼저 인지. system prompt 분리가 올바른 방식임을 직접 판단한 뒤 AI에게 서버 설계를 지시. README에 수정 이유 작성 |
| 스타일 선택기 도입 시 테스트 실패 | `sendMessage(sessionId, content, style)` 3개의 인자 호출로 구현 | 테스트 실패 원인 직접 파악 후 수정 지시 | 기존 테스트가 `mockSendMessage`를 2개의 인자로 기대하는 구조를 분석해 style이 `undefined`일 때는 인자를 아예 전달하지 않는 spread 방식으로 수정 지시 |

## AI 없이 직접 판단한 부분

- **답변 스타일 기능의 구현 방식**: 클라이언트 단독 구현(message prefix 방식) vs 서버 수정(system prompt 방식) 중 서버 수정이 필요하다고 먼저 판단한 뒤 AI에게 설계를 요청. AI는 처음에 서버 수정 없이 prefix를 붙이는 방안을 언급했으나, loadSession 후 prefix가 노출된다는 문제를 내가 먼저 인지함
- **커밋 단위 설계**: 기능 단위 커밋 전략을 직접 설계하고, AI가 제안한 커밋 메시지는 매번 검토 후 승인
- **UI 완성도 최종 판단**: 모든 시각적 결과물은 직접 브라우저에서 확인 후 승인 또는 수정 지시

## 기타 메모

Claude Code를 주력 도구로 사용해 대부분의 코드를 생성했다. 설계 수준의 결정(훅 분리, 서버 수정 여부, 아키텍처 방향)은 AI와 대화를 통해 방향을 잡고, 실제 코드는 AI가 작성하는 방식으로 진행했다.

테스트 호환성 문제는 직접 발견하고 방향을 잡았다. 스타일 선택기 구현 후 기존 테스트가 `sendMessage` 2개의 인자를 기대하는 구조임을 직접 파악하고, style이 없을 때는 인자를 아예 전달하지 않는 방식으로 수정을 지시했다. AI는 spread trick(`...(style ? [style] : [])`)으로 구현했다.

반면 **시각적 판단이 필요한 UI 작업**은 AI만으로는 완결되지 않았다. 컴포넌트 위치나 여백 같은 부분은 실제로 브라우저에서 확인한 뒤 수정 지시를 내려야 했고, 마크다운 렌더링이 적용되지 않는다는 것도 직접 테스트해서 발견했다. AI는 코드 작성에는 강하지만 시각적 결과물의 품질은 사람이 직접 검증해야 한다는 것을 다시 확인했다.

이번 작업에서는 `PLAN.md`를 직접 작성해 전체 구현 흐름과 커밋 히스토리를 하나의 문서로 관리했다. 컨텍스트를 잘 정리해두는 것만으로도 AI에게 불필요한 설명을 줄이고 더 정확한 응답을 이끌어낼 수 있었으며, 토큰 낭비도 줄일 수 있었다. Claude Code에는 `/memory` 기능을 통해 컨텍스트를 자동으로 저장하는 기능이 추가되었지만, 복잡한 프로젝트일수록 도메인별 또는 기능별로 컨텍스트를 직접 구조화해서 관리하는 방식이 더 정밀한 협업을 가능하게 한다고 생각한다.
